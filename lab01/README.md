# Smart Device Manager — Паттерн Prototype

> Учебный проект на C# (Windows Forms), демонстрирующий применение порождающего паттерна проектирования **Prototype**.

---

## 1. Описание проблемы

**Предметная область:** Графический редактор устройств умного дома.

Приложение позволяет пользователю размещать на интерактивном холсте устройства умного дома двух типов — **SmartCamera** (умная камера) и **SmartLight** (умная лампа)

Каждое устройство имеет набор уникальных свойств:
- `SmartCamera` — разрешение, ночной режим, угол обзора, координаты
- `SmartLight` — яркость, цветовая температура, состояние, координаты

В процессе работы возникает задача: **создать точную копию уже настроенного объекта** с сохранением всех его параметров. Например, пользователь настроил камеру с нужным разрешением и углом — и хочет разместить такую же в другом месте холста.

Наивное решение — копировать поля вручную прямо в коде формы:
```csharp
if (selectedDevice is SmartCamera cam)
    devices.Add(new SmartCamera(cam.X + 20, cam.Y + 20, cam.Resolution, cam.NightMode, cam.Angle));
else if (selectedDevice is SmartLight light)
    devices.Add(new SmartLight(light.X + 20, light.Y + 20, light.Brightness, light.ColorTemp, light.State));
```

**Проблемы этого подхода:**
- Форма знает о внутренних полях каждого класса — нарушается инкапсуляция
- При добавлении нового поля в класс нужно обновлять код формы
- При добавлении нового типа устройства — расширять цепочку `if/else`
- Логика копирования размазана по коду и дублируется

---

## 2. Решение: паттерн Prototype

**Prototype** — порождающий паттерн проектирования, который делегирует создание копии самому объекту через метод `Clone()`. Клиент (форма) не знает ничего о внутреннем устройстве объекта — он просто вызывает `Clone()` через интерфейс.

### Интерфейс ISmartDevice
```csharp
public interface ISmartDevice
{
    ISmartDevice Clone();
    void Draw(Graphics g);
    string GetInfo();
    bool ContainsPoint(int x, int y);
    int X { get; set; }
    int Y { get; set; }
}
```

### Реализация Clone() в SmartCamera
```csharp
public ISmartDevice Clone()
{
    return new SmartCamera(
        this.X + 20,
        this.Y + 20,
        this.Resolution,
        this.NightMode,
        this.Angle
    );
}
```

### Реализация Clone() в SmartLight
```csharp
public ISmartDevice Clone()
{
    return new SmartLight(
        this.X + 20,
        this.Y + 20,
        this.Brightness,
        this.ColorTemp,
        this.State
    );
}
```

### Вызов в форме
```csharp
// Форма не знает, камера это или лампа — просто вызывает Clone()
ISmartDevice clone = selectedDevice.Clone();
devices.Add(clone);
```

## 4. Вывод

Внедрение паттерна **Prototype** положительно повлияло на архитектуру приложения:

| Аспект | Без паттерна | С паттерном |
|---|---|---|
| Инкапсуляция | Нарушена — форма знает о полях | Соблюдена — форма работает через интерфейс |
| Добавление нового поля | Нужно менять код формы | Меняется только сам класс |
| Добавление нового типа устройства | Расширяем `if/else` в форме | Реализуем `Clone()` в новом классе |
| Объём кода копирования | Растёт с каждым типом | Не растёт |
| Принцип открытости/закрытости | Нарушен | Соблюдён |

Паттерн Prototype позволил:
- Убрать зависимость `Form1` от конкретных реализаций устройств
- Сосредоточить логику копирования внутри каждого класса
- Упростить добавление новых типов устройств — достаточно реализовать интерфейс `ISmartDevice`
- Сократить код формы и избавиться от цепочки `if/else` при дублировании

Важно отметить, что паттерн Prototype является **архитектурным решением** и никак не влияет на внешний вид и поведение программы с точки зрения пользователя. Конечный пользователь не заметит никакой разницы — кнопка «Дублировать» работает одинаково как с паттерном, так и без него. Разница проявляется исключительно на уровне кода: в его читаемости, расширяемости и соблюдении принципов объектно-ориентированного проектирования.
